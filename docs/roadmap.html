AuditPRO V2 - Roadmap Técnico & Especificação

Data: 28 de Janeiro de 2026

Este documento descreve a arquitetura técnica e o plano de implementação para transformar a ferramenta atual em um "Assistente de Qualidade de Dados" completo.

1. Resumo Executivo

O backend atual, construído em Python (Flask) e Supabase (PostgreSQL), fornece uma base sólida. A maioria das novas features requer expansão do schema do banco de dados e integração de bibliotecas Python específicas (scikit-learn, APScheduler).

2. Análise de Gap & Atualizações de Arquitetura

2.1 Dashboard & Visualização

  • Estado Atual: Tabela analysis_history existe.
  • Mudanças Necessárias:
    • Criar endpoint /api/analytics/dashboard para agregar dados históricos.
    • Frontend: Instalar Chart.js. Implementar views de tendência.

2.2 Machine Learning (Detecção de Anomalias)

  • Estado Atual: Inexistente.
  • Mudanças Necessárias:
    • Novo módulo src/ml_engine.py usando scikit-learn (IsolationForest).
    • Nova tabela ml_models para armazenar modelos treinados.

2.3 IA Avançada (Chat with Data)

  • Estado Atual: LangChain integrado.
  • Mudanças Necessárias:
    • RAG: Ingestão de metadados e resultados no Supabase pgvector.
    • Endpoint de Chat para perguntas em linguagem natural sobre o dataset.

2.4 Automação & Agendamento

  • Estado Atual: Execução manual.
  • Mudanças Necessárias:
    • Integrar APScheduler.
    • Nova tabela scheduled_jobs.

2.5 Gestão de Workflow

  • Estado Atual: Queries salvas com status básico.
  • Mudanças Necessárias:
    • Adicionar colunas: assigned_to, resolution_notes.
    • Endpoints para atribuição e resolução.

2.6 Segurança Corporativa (RBAC)

  • Estado Atual: Modelo de dono único.
  • Mudanças Necessárias:
    • Nova tabela project_members com papéis (Admin, Editor, Leitor).
    • Atualizar queries para respeitar permissões de membro.

3. Plano de Implementação

Fase 1: Workflow e Visualização (Valor Imediato) Implementação dos Dashboards e do sistema de Atribuição/Resolução de queries. É a funcionalidade mais visível para o usuário final.
Fase 2: Automação Configuração do APScheduler e sistema de Alertas por Email. Garante recorrência do uso.
Fase 3: Deep Tech (ML & IA) Treinamento dos modelos de Isolation Forest e implementação do Chat com Dados (Vector Search).
Fase 4: Enterprise (RBAC & Audit) Refatoração para múltiplos usuários por projeto e logs imutáveis (Compliance).

4. Script de Migração (Supabase)

-- 1. Workflow de Queries
ALTER TABLE saved_queries 
ADD COLUMN assigned_to UUID REFERENCES auth.users(id),
ADD COLUMN resolution_notes TEXT,
ADD COLUMN history JSONB DEFAULT '[]';

-- 2. Agendamento
CREATE TABLE scheduled_jobs (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    project_id UUID REFERENCES projects(id),
    schedule_type TEXT CHECK (schedule_type IN ('daily', 'weekly')),
    next_run TIMESTAMP WITH TIME ZONE,
    is_active BOOLEAN DEFAULT true,
    recipients JSONB
);

-- 3. Controle de Acesso (RBAC)
CREATE TABLE project_members (
    project_id UUID REFERENCES projects(id),
    user_id UUID REFERENCES auth.users(id),
    role TEXT CHECK (role IN ('owner', 'editor', 'viewer')),
    PRIMARY KEY (project_id, user_id)
);

-- 4. ML Models
CREATE TABLE ml_models (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    project_id UUID REFERENCES projects(id),
    model_type TEXT,
    serialized_model BYTEA
);
